2 Git 三区域模型：为什么 checkout 是「一秒恢复」

Working Directory
你看到的文件
git checkout → 覆盖这里
Staging Area (Index)
git add 放这里
暂存区也一并切换
Repository (.git)
所有 commit 快照
checkout 30ms 完成

关键机制：Git 不"修改"文件来切换分支——它直接替换整个 Working Directory 的快照。 .git/objects 中存储了每个 commit 的完整文件树，git checkout 只是把另一个快照解压到工作目录。 这就是为什么从 failure 分支切回 main 不到 1 秒就能完全恢复。

❌ failure 分支：返回 Array
serialize(): FileNode[] {
  const result: FileNode[] = [];
  for (const [_, node] of this.files) {
    result.push({ type, name, path, content });
  }
  return result;  // ["0":{...}, "1":{...}]
}
// Object.keys(["0","1"]) → ["0","1"]
// → 文件路径变成 /0, /1, /2...
✅ main 分支：返回 Record
serialize(): Record<string, FileNode> {
  const result: Record<string, FileNode> = {};
  for (const [path, node] of this.files) {
    result[path] = { type, name, path, content };
  }
  return result;  // {"/App.jsx":{...}}
}
// Object.keys({"/App.jsx":...}) → ["/App.jsx"]
// → 正确的文件路径
传导链： serialize() 改返回值 → 存 DB 格式变 → 刷新时 deserializeFromNodes() 解析错误 → FileSystemContext 状态损坏 → PreviewFrame 找不到入口文件 → 白屏

4 CLAUDE.md 规则外化：一句话省三轮对话

- No disk writes — paths don't exist on disk
- Use VFS.rename(), NOT fs.renameSync()

为什么是「一句话省三轮」？
AI 不知道哪些设计决策是「反直觉」的。fs.renameSync 对任何有 Node 经验的开发者都是本能选择。 如果不写在 CLAUDE.md，每次 AI 写文件操作都会踩坑 → 你纠正 → AI 换方案 → 你纠正 → 反复三轮。 一条规则写进 CLAUDE.md，所有后续对话受益。

6 Contract-First：多 Agent 的 API 合同

三个 Agent 各写各的，三处不匹配

Agent角色写的代码假设的合同
Agent A 前端 UI ShareButton.tsx POST /api/shares + {project_id} + 响应 {id, url}
Agent B 后端 API route.ts POST /api/share + {projectId} + 响应 {shareId, shareUrl}
Agent C 集成 连不上 不知道 A 和 B 没对齐
Contract-First 解法：
  1. 先定义 API 合同文档（请求 Method/Path/Body + 响应结构）
  2. 三个 Agent 都对着同一份合同写代码
  3. 合同即验收标准 — 不符合合同的代码 = bug

类比建筑：三个施工队各拿不同的图纸盖同一栋楼 → 楼梯对不上走廊。 Contract-First = 先对齐图纸，再各干各的。

7 总结：四条防线的协同

防线对应章节防护的 Bug 类型反馈速度
Git 护栏 二 任何代码改动都能秒级回退 30 秒（checkout）
CLAUDE.md 规则外化 三 AI 不理解的反直觉设计决策 写一次，永久生效
类型系统 + 测试 四 跨模块的类型契约破坏 build 时（tsc）/ test 时
Contract-First 七 多 Agent 间的接口不一致 写合同阶段（写代码前）

从修复速度看：Contract-First（防患于未然） → Git（秒级恢复） → 类型系统（编译时） → 测试（运行时）
